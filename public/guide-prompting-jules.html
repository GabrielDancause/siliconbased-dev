The Big Discovery: Jules Can Scrape the Web

We assumed Jules couldn't make outbound HTTP requests from its sandbox. We were wrong. Jules runs in a GitHub Codespace environment that has full internet access. It can:

  • Make HTTPS GET/HEAD requests to any public URL
  • Parse HTML responses and extract data
  • Read JSON-LD structured data from web pages
  • Follow redirects and handle timeouts
  • Process responses and save results to JSON files

This changed everything. Instead of Jules only building static tool pages, we could use it to produce original research — crawl real websites, analyze real data, and build interactive study pages.

Lesson 1: Tell Jules What It CAN Do

Key Insight
Jules defaults to the conservative path unless told otherwise

If you don't explicitly say "you can make HTTP requests," Jules will assume it can't and will hardcode fake data instead. Always start your prompt with capabilities.

❌ Before
"Create a study analyzing 100 websites' page weight"

Result: Jules hardcoded fake data or said it couldn't make requests
✅ After
"You CAN make outbound HTTP requests. This task requires you to actually crawl real websites."

Result: Jules made real HTTP requests and collected real data

Lesson 2: Define the Exact Methodology

Key Insight
Don't let Jules figure out HOW to measure — tell it

Our first page weight study measured only the HTML response body (50-200KB per site). That's not "page weight" — real page weight includes CSS, JS, images, and fonts (typically 2-5MB). Jules did exactly what we asked; we just asked wrong.

❌ Vague
"Measure the page weight of each site"

Result: Measured HTML response only (17KB for google.com)
✅ Specific
"Page weight = HTML + all CSS + all JS + images. Parse HTML for resource URLs, make HEAD requests for Content-Length on each, sum everything."

Result: Full resource measurement (1.2MB for google.com)

Lesson 3: Give It the Data Schema

Key Insight
Provide the exact JSON structure you want back

When we didn't specify field names, Jules used different names than expected. The page template then couldn't read the data.

{
  "crawlDate": "ISO date",
  "results": [
    {
      "domain": "google.com",      // exact field name
      "htmlBytes": 17000,           // exact field name
      "totalBytes": 532000,         // computed sum
      "error": null                 // null, not undefined
    }
  ]
}

Include a complete example object with comments. Jules follows the schema precisely when given one.

Lesson 4: Anti-Hallucination Instructions

Key Insight
"If you only get 50, say 50. Don't fake 500."

AI coding agents will try to meet your target. If you ask for 500 recipes and they can only scrape 50, they might pad the data. Explicitly tell them that honest counts are better than fabricated data.

Include these exact phrases in your prompt:

  • "null over fake data, always"
  • "Be honest about the count — the title must match actual data"
  • "If you can't determine a value, set it to null — do NOT guess"

Lesson 5: The Close → Reprompt Reflex

Key Insight
First attempt rarely perfect. Close the PR, tell Jules what went wrong, send again.

This became our default workflow. Every study improved on the second attempt because we could tell Jules exactly what went wrong.

  1. Dispatch study → Jules builds it in ~30 minutes
  2. Review the PR → Check data quality, not just if it builds
  3. Close with feedback → "Closing: hydration calculation is wrong, only 25 recipes"
  4. Reprompt with fixes → "Your previous attempt had these problems: [specifics]"
  5. Review v2 → Usually good enough to merge

Lesson 6: Domain Knowledge Must Be Explicit

Key Insight
Jules doesn't know that 100% hydration sourdough is ciabatta, not standard bread

You must encode domain knowledge into the prompt, including what "normal" looks like.

❌ No context
"Calculate hydration from ingredients"

Result: 100% hydration for standard loaves (wrong)
✅ With domain knowledge
"Hydration = water ÷ flour × 100. Normal: 60-90%. If >90%, flag as high. If >100%, likely error — set to null."

Result: Average 77.8% (correct)

Lesson 7: Data Validation Ranges

Key Insight
Give explicit valid ranges for every numeric field

Without validation rules, Jules includes outliers that are clearly errors (oven temp of 0°F, page weight of -1 bytes).

Data validation:
- hydration: 55-100 (flag >90), else null
- ovenTempF: 300-550, else null
- bakeTimeMin: 15-120, else null
- apyPct: 0.01-6.00, else null

Lesson 8: Visual Style Matching

Key Insight
Reference an existing file and give exact CSS values

"Match the site style" isn't enough. Give exact hex codes, font names, and reference a specific file.

Style: bg #060a12, cards #0a1020, border #152040
Accent: #4a8fe7 (Westmount) or #6366f1 (PhotonBuilder)
Fonts: Inter (body) + JetBrains Mono (code)

Lesson 9: Listing Pages > Individual Pages

Key Insight
Point Jules at index/search pages, not individual URLs

Point at listing pages that link to dozens of items, not individual pages one at a time.

Don't: "Scrape recipes from kingarthurbaking.com"
Do:   "Fetch the listing page at /recipes/sourdough
       — extract all recipe URLs, then fetch each one."

Lesson 10: Multi-Domain Repo Routing

Key Insight
Each domain has its own repo — route to the correct one or everything lands in the wrong place

When running a factory across multiple domains, every study must target the correct GitHub repo. Jules sessions can't be redirected mid-build.

DOMAIN_TO_REPO = {
    'siliconbased.dev':          'siliconbased-dev',
    'westmountfundamentals.com': 'westmount-research',
    'firemaths.info':            'firemaths-info',
    '28grams.vip':               '28grams-vip',
    ...
}

Lesson 11: Always Update the Homepage

Key Insight
Jules builds the study page perfectly — then forgets the homepage exists

If the homepage never gets updated, nobody discovers the study. Add explicit instructions to update index.astro with every new page.

Reality check: Jules updates the homepage maybe 30% of the time, even when asked. This is why we now handle it in post-merge validation (see Lesson 19).

Lesson 12: Know the Homepage Structure

Key Insight
4 sections — Tools, Studies, Guides, Lists — each with different card formats

Every homepage follows the same 4-section layout. Tell Jules which section its page belongs to and the exact card format.

guide-*.html  → GUIDE section
list-*.html   → LIST section
*-study.html  → STUDY section
everything else → TOOL section

Lesson 14: Auto-Finalize Blocked Sessions

Key Insight
When Jules asks a question, don't answer — just tell it to ship

POST /sessions/{id}:sendMessage with empty body {}. Jules finalizes immediately. Review the PR afterward — if it's bad, close and reprompt.

Lesson 15: The Zero-Click SEO Problem NEW — March 11

Critical Discovery
Google showed us impressions but zero clicks — because we answered the query in the snippet

We saw our pages ranking just below the AI Overview on Google. Hundreds of impressions. Zero clicks. The problem: our meta descriptions gave away the answer. The user got what they needed without ever visiting our site.

❌ Zero-Click Snippet
"What is the average ETF expense ratio? Across our index of 86 major ETFs, the average expense ratio is 0.164% and the median is 0.09%."

Result: User got the answer → closed tab → zero clicks
✅ Curiosity Gap
"Some popular ETFs charge 10x more than nearly identical alternatives. We ranked 86 funds to find out which ones are quietly overcharging you."

Result: User wants to know WHICH ones → clicks through

The pattern: [Surprising hook] + [Scope of research] + [Reason to click]

We rewrote all 40 meta descriptions on Westmount Fundamentals in one batch. This is now a mandatory requirement in every Jules prompt.

## META DESCRIPTION RULE
NEVER reveal the key stat or answer in the meta description.
Create a curiosity gap. Tease the data, make them click.

❌ "The S&P 500 has 37.3% in just 10 stocks"
✅ "Just 10 stocks now control over a third of the S&P 500.
   See the full weight rankings and how concentrated 
   your index fund really is."

Lesson 16: Brand Consistency at Scale NEW — March 11

Key Insight
Jules uses random accent colors unless you specify hex codes — and even then, gets it wrong 20% of the time

Our first batch of automated PRs had pages with green, orange, and teal accents — on a site that uses indigo (#6366f1). Wrong brand color = immediate PR rejection.

Per-domain accent colors we enforce:

Westmount Fundamentals: #4a8fe7 (blue)
PhotonBuilder:          #6366f1 (indigo)
siliconbased.dev:       #818cf8 (purple)
firemaths.info:         #D4A017 (gold)
migratingmammals.com:   #C4956A (tan)
aliimperiale.com:       #AD9846 (gold)
28grams.vip:            #C2185B (pink)

Include the hex code in the prompt AND check it during PR review. The validation script (Lesson 17) could be extended to verify this automatically.

Lesson 17: Post-Merge Validation Scripts NEW — March 11

Game Changer
A Python script that catches every missing element and auto-fixes what it can

Instead of manually checking each merged page, we built a validator that runs after every merge batch. It checks 10 things and auto-fixes 3 of them.

What the validator checks

  1. ✅ Google Analytics tag (correct per-domain ID)
  2. ✅ Unified nav bar (sticky, with section links)
  3. ✅ Meta description exists
  4. ⚠️ Meta description doesn't give away the answer
  5. ⚠️ FAQPage JSON-LD schema present
  6. ✅ Mobile viewport meta tag
  7. ❌ Listed on homepage (index.astro)
  8. ❌ Present in sitemap.xml
  9. ✅ No old/wrong GA tags
  10. ✅ No "Built on PhotonBuilder" nav branding

What it auto-fixes (with --fix flag)

  • Missing GA tag → Injects before </head>
  • Missing nav bar → Injects after <body>
  • Wrong GA tag → Replaces with correct domain-specific ID
python3 validate-westmount.py --fix

📄 guide-technical-analysis-basics.html
   ❌ Missing unified nav bar
   ✅ Fixed: injected nav bar
   ❌ Not listed on homepage (index.astro)

============================================================
Pages checked: 43
Issues found: 3
Auto-fixed: 1
  Errors: 2
  Warnings: 1

The workflow: Jules builds the page (~90% correct) → validator catches the remaining 10% → you manually add to homepage + regenerate sitemap → push.

Lesson 18: The Mandatory Prompt Template NEW — March 11

Key Insight
Every Jules prompt MUST include 11 mandatory requirements — the "rejection checklist" evolved

After seeing what Jules consistently misses, we codified all requirements into a numbered checklist that goes at the END of every prompt.

MANDATORY REQUIREMENTS (PR will be rejected if ANY are missing):

1. FILE: public/{slug}.html — self-contained, ALL CSS/JS inline
2. NAV BAR: Sticky nav with brand name, links to /#studies /#tools /#guides
3. GOOGLE ANALYTICS: gtag with domain-specific ID before </head>
4. META DESCRIPTION: Curiosity gap — NEVER reveal the key stat
5. FAQ SECTION: 5 Q&As matching real search queries
6. JSON-LD: FAQPage schema with all 5 Q&As
7. DESIGN: Correct accent color, dark theme, Inter + JetBrains Mono
8. FOOTER: © 2026 [Brand] · A GAB Ventures property + disclaimer
9. MOBILE: viewport meta tag, responsive design
10. SITEMAP: Update sitemap.xml
11. CANONICAL: <link rel="canonical"> with full URL

Lesson 19: Homepage Visibility Debt NEW — March 11

Hard Data
Westmount had 40 pages but only 17 on the homepage — more than half were invisible

Pages that aren't on the homepage don't get discovered. They exist, they rank on Google, but nobody browsing the site can find them. We had accumulated 23 "invisible" pages over several weeks of merging.

The fix: After every merge batch, check the homepage against the filesystem. The validator flags pages not listed on the homepage. Manual step: add them to the correct section in index.astro.

Prevention: Jules can't reliably edit Astro files. Accept this limitation and handle homepage updates as a human/automation step post-merge.

Lesson 20: Jules PRs Silently Fail NEW — March 11

Gotcha
Jules completes the session and builds the code — but sometimes never creates the PR

We dispatched 4 sessions with automationMode: AUTO_CREATE_PR. All 4 completed successfully. Zero PRs were created. No error. The patches were in the session outputs but never pushed to GitHub.

The workaround: Check session outputs for changeSet.gitPatch.unidiffPatch. If there's a patch but no pullRequestUrl, extract the patch and git apply it manually.

curl -s "sessions/$SID" -H "X-Goog-Api-Key: $KEY" | \
  python3 -c "import sys,json
d = json.load(sys.stdin)
for o in d.get('outputs', []):
    patch = o.get('changeSet',{}).get('gitPatch',{}).get('unidiffPatch','')
    if patch:
        open('/tmp/patch.diff','w').write(patch)
        print(f'Saved {len(patch)} bytes')"

cd ~/repo && git apply /tmp/patch.diff

This happened in ~30% of our sessions. Still investigating if it's a Jules bug or a race condition with branch creation.

Lesson 21: GA Tags Per Domain NEW — March 11

Operational
Don't share one GA property across all domains — you'll lose per-site analytics

We initially used one GA tag (G-G86C7NJG3F) across all 9 domains. This made it impossible to see which domain was getting traffic. Now each domain has its own property.

GA IDs per domain:
westmountfundamentals.com  →  G-VYF72NSC1Q
aliimperiale.com           →  G-23G2Y8PVLR
All other domains          →  G-G86C7NJG3F

Gotcha: If the site uses Astro, the GA tag in BaseHead.astro (layout) and the tags in public/*.html (standalone pages) are separate. Update BOTH or Google verification will fail on the homepage while individual pages work fine.

Lesson 22: Unified Nav Injection NEW — March 11

Key Insight
Astro-rendered pages have a Nav component — standalone HTML pages in public/ do not

Only 6 out of 40 pages had any nav bar at all. The ones that did had inconsistent links and branding. Solution: inject a unified inline nav into every standalone HTML page.

The nav is pure inline HTML + inline styles (no external CSS needed). It's injected right after <body> either by Jules (in the prompt template) or by the validation script (auto-fix).

Rule for sub-sites (PhotonBuilder): Never put "Built on PhotonBuilder" or "← PhotonBuilder" in the nav. Footer credit only. Nav branding screams "template."

Lesson 23: Sub-site Index Pages NEW — March 11

Quick Fix
GitHub Pages doesn't auto-generate directory listings — no index.html = 404

We created /auberge/phone-answering-service-research.html but visiting /auberge/ gave a 404. Every sub-directory needs its own index.html that serves as a hub page.

Lesson 24: The Full Automated Process (March 2026) NEW

The System
Three-stage pipeline: Dispatch → Review → Validate

Stage 1: Dispatch (Content Factory)

A cron job dispatches 12 Jules sessions every 30 minutes — 6 per domain. Each prompt includes the full 11-point rejection checklist, correct GA tag, nav bar HTML, accent color, and curiosity-gap rule.

Every 30 min:
1. Check existing pages (avoid duplicates)
2. Check open PRs (avoid duplicates)
3. Pick 6 high-value pages per domain
4. Dispatch via Jules API with full template
5. Auto-finalize any AWAITING_USER_FEEDBACK sessions

Stage 2: Review (Review Factory)

A separate cron reviews completed PRs every 30 minutes. Merges good ones, cherry-picks conflicts, closes bad ones.

Every 30 min (offset by 15 min):
1. List open PRs on both repos
2. For each: check files, spot-check content, verify accent color
3. Good PRs → merge (or cherry-pick if conflict)
4. Bad PRs → close with comment
5. Regenerate sitemaps after merging

Stage 3: Validate (Post-Merge)

After merging, run the validation script. It catches and auto-fixes missing GA tags, nav bars, and wrong GA tag IDs. Then manually update the homepage and push.

After merge:
1. python3 validate-westmount.py --fix
2. Add new pages to index.astro (correct section)
3. git add, commit, push
4. Verify on live site
