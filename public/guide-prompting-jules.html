The Big Discovery: Jules Can Scrape the Web

We assumed Jules couldn't make outbound HTTP requests from its sandbox. We were wrong. Jules runs in a GitHub Codespace environment that has full internet access. It can:

  • Make HTTPS GET/HEAD requests to any public URL
  • Parse HTML responses and extract data
  • Read JSON-LD structured data from web pages
  • Follow redirects and handle timeouts
  • Process responses and save results to JSON files

This changed everything. Instead of Jules only building static tool pages, we could use it to produce original research — crawl real websites, analyze real data, and build interactive study pages.

Lesson 1: Tell Jules What It CAN Do

Key Insight
Jules defaults to the conservative path unless told otherwise

If you don't explicitly say "you can make HTTP requests," Jules will assume it can't and will hardcode fake data instead. Always start your prompt with capabilities.

❌ Before
"Create a study analyzing 100 websites' page weight"

Result: Jules hardcoded fake data or said it couldn't make requests
✅ After
"You CAN make outbound HTTP requests. This task requires you to actually crawl real websites."

Result: Jules made real HTTP requests and collected real data

Lesson 2: Define the Exact Methodology

Key Insight
Don't let Jules figure out HOW to measure — tell it

Our first page weight study measured only the HTML response body (50-200KB per site). That's not "page weight" — real page weight includes CSS, JS, images, and fonts (typically 2-5MB). Jules did exactly what we asked; we just asked wrong.

❌ Vague
"Measure the page weight of each site"

Result: Measured HTML response only (17KB for google.com)
✅ Specific
"Page weight = HTML + all CSS + all JS + images. Parse HTML for resource URLs, make HEAD requests for Content-Length on each, sum everything."

Result: Full resource measurement (1.2MB for google.com)

Lesson 3: Give It the Data Schema

Key Insight
Provide the exact JSON structure you want back

When we didn't specify field names, Jules used different names than expected (name instead of recipeName, source instead of sourceDomain). The page template then couldn't read the data.

{
  "crawlDate": "ISO date",
  "methodology": "description of what was measured",
  "totalSites": 100,
  "results": [
    {
      "domain": "google.com",      // exact field name
      "htmlBytes": 17000,           // exact field name
      "cssBytes": 45000,            // exact field name
      "jsBytes": 320000,            // exact field name
      "totalBytes": 532000,         // exact field name
      "error": null                 // null, not undefined
    }
  ]
}

Include a complete example object with comments. Jules follows the schema precisely when given one.

Lesson 4: Anti-Hallucination Instructions

Key Insight
"If you only get 50, say 50. Don't fake 500."

AI coding agents will try to meet your target. If you ask for 500 recipes and they can only scrape 50, they might pad the data. Explicitly tell them that honest counts are better than fabricated data.

Include these exact phrases in your prompt:

  • "Be honest about the count — the title must match actual data"
  • "null over fake data, always"
  • "If you can't determine a value, set it to null — do NOT guess"
  • "Better to have 50 real data points than 500 fake ones"

Lesson 5: The Close → Reprompt Reflex

Key Insight
First attempt rarely perfect. Close the PR, tell Jules what went wrong, send again.

This became our default workflow. Every study improved on the second attempt because we could tell Jules exactly what went wrong:

  1. Dispatch study → Jules builds it in ~30 minutes
  2. Review the PR → Check data quality, not just if it builds
  3. Close with feedback → "Closing: hydration calculation is wrong, only 25 recipes, need 100+"
  4. Reprompt with fixes → "Your previous attempt had these problems: [specifics]. Fix them."
  5. Review v2 → Usually good enough to merge

Our sourdough study went from 25 recipes with wrong hydration (v1) to 85 recipes with correct hydration (v2). The page weight study went from HTML-only (v1) to full resource measurement (v2).

Lesson 6: Domain-Specific Knowledge Must Be Explicit

Key Insight
Jules doesn't know that 100% hydration sourdough is ciabatta, not standard bread

When Jules calculated sourdough hydration and got 100%, it didn't flag it as unusual — because it doesn't know baking. You must encode domain knowledge into the prompt.

❌ No domain context
"Calculate hydration from ingredients"

Result: 100% hydration for standard loaves (wrong calculation)
✅ With domain knowledge
"Hydration = water weight ÷ flour weight × 100. Normal range: 60-90%. If result > 90%, flag as high hydration. If > 100%, likely a calculation error — set to null."

Result: Average hydration of 77.8% (correct)

Lesson 7: Data Validation Ranges

Key Insight
Give explicit valid ranges for every numeric field

Without validation rules, Jules will include outliers that are clearly errors (oven temp of 0°F, page weight of -1 bytes). Define the acceptable range for every field.

Data validation:
- hydration: 55-100 (flag >90 as "high hydration"), else null
- ovenTempF: 300-550, else null
- bakeTimeMin: 15-120, else null
- apyPct: 0.01-6.00, else null
- costUSD: 0-5000, else null

Lesson 8: Visual Style Matching

Key Insight
Reference an existing file and give exact CSS values

Saying "match the site style" isn't enough. Give exact hex codes, font names, border-radius values, and reference a specific existing file Jules can read from the repo.

Style (match existing security-headers-study.html):
- Background: #0a0b10
- Cards: #111318 with border #1e2030, border-radius 14px
- Accent: #818cf8
- Fonts: Inter (body) + JetBrains Mono (code) from Google Fonts
- Text: #e0e0e0 body, #999 secondary, #666 tertiary

Lesson 9: Listing Pages > Individual Pages

Key Insight
Point Jules at index/search pages, not individual recipe URLs

Our first sourdough attempt got 25 recipes because Jules only scraped individual pages. The v2 prompt pointed it at listing pages (recipe indexes, search results) that link to dozens of recipes each.

Don't: "Scrape sourdough recipes from kingarthurbaking.com"
Do:   "Fetch https://www.kingarthurbaking.com/recipes/sourdough
       — this is a listing page with 50+ recipe links.
       Extract all recipe URLs, then fetch each one."

The Full Research Workflow

Step 1: Choose a Study

Pick something where the data exists publicly but nobody has packaged it into a single comparison. The value is in the aggregation.

Step 2: Write the Prompt

Use the template below. Include: capabilities declaration, source URLs, methodology, data schema, validation rules, output file specs, visual style, anti-hallucination rules.

Step 3: Dispatch via API

POST https://jules.googleapis.com/v1alpha/sessions
{
  "prompt": "...",
  "sourceContext": {
    "source": "sources/github/OWNER/REPO",
    "githubRepoContext": {"startingBranch": "main"}
  },
  "automationMode": "AUTO_CREATE_PR",
  "title": "Research: Study Name"
}

Step 4: Review (~30 min later)

Check the PR. Verify: Is the data real? Are the calculations correct? Are there enough data points? Is the methodology honest?

Step 5: Close → Reprompt or Merge

If the data is wrong or insufficient, close the PR with specific feedback and dispatch again. If it's good enough, merge. You can always improve in a future session.

What Jules Can't Do

  • No headless browser — can't render JavaScript-heavy SPAs or measure real paint times
  • No login/auth — can only access public pages
  • Anti-scraping — some CDNs block requests from Codespace IPs; expect 10-20% failure rate
  • No physical measurement — can't taste food, test hardware, or run lab experiments
  • Rate limits — 300 sessions/day per Google account, 15 concurrent max
  • Clarification UI-only — if Jules asks a question, you can only answer in the web UI, not via API
  • One repo per session — but you can target any repo your account has access to

What Jules IS good at: Compiling, scraping, and packaging public data that exists across many sources into a single structured, searchable resource. The value is in the aggregation, not the data itself.

The Prompt Template

Copy this structure for any new research study:

RESEARCH STUDY: "[Title]" for [domain]

You CAN make outbound HTTP requests. Use them.

## What this study is
[One paragraph explaining the study and why it matters]

## How to collect the data

### Sources to scrape
1. [URL] — [description]
2. [URL] — [description]
[List 5-10 source URLs]

### Data to extract per [item]
- **fieldName**: description (type)
[List every field with types]

### Data validation
- fieldName: [min]-[max], else null
[Validation for every numeric field]

## Output files

### 1. scripts/crawl-[topic].cjs
Node.js CommonJS. [Scraping instructions]. 500ms delays.

### 2. data/[topic]-study.json
[Exact JSON schema with example object]

### 3. public/[topic]-study.html
Self-contained. ALL inline CSS+JS. NO frameworks.
Style: bg #0a0b10, cards #111318, borders #1e2030,
accent [hex], Inter + JetBrains Mono.
[Page structure: hero, stats, charts, findings, table,
methodology, citation, footer, JSON-LD]

### 4. package.json + deploy.yml
[Script name and workflow step]

## CRITICAL
- Get at least [N] items. Aim for [M]+.
- null over fake data, always
- Be honest about count — title must match
- [Domain-specific validation rules]
