What is a good API response time?
A "good" response time depends on the use case. For user-facing web applications, API responses under 100ms feel instantaneous, while anything between 100ms to 300ms is generally acceptable. Background tasks or batch processing can tolerate much higher latency (500ms+). Anything over 1000ms (1 second) usually requires loading states or spinners in the UI to prevent user frustration.
Which free APIs are the most reliable?
Utility APIs backed by major cloud providers or large CDNs tend to be the most reliable. In our tests, endpoints like GitHub API, JSONPlaceholder, and Cloudflare-backed services consistently return successful status codes with low variance in latency. Hobbyist or community-run APIs often experience occasional timeouts or 5xx errors under load.
Do free APIs have rate limits?
Yes, almost all free public APIs implement rate limiting to prevent abuse. Rate limits can range from very strict (e.g., 60 requests per hour for unauthenticated GitHub API access) to very generous (e.g., 10,000 requests per day for Open Meteo). Some APIs, like JSONPlaceholder, do not have strict published limits but will block abusive IP addresses. Always check the API's documentation and send requests responsibly.
How do you measure API latency?
Latency is measured as the total time elapsed from initiating the HTTP GET request to fully receiving the response body and the TCP connection closing. In Node.js, this is calculated using Date.now() before the request starts and after the end event fires on the response stream. This "Round Trip Time" (RTT) includes DNS resolution, TCP handshake, TLS negotiation, server processing time, and network transfer time.

Methodology

Sample Size: Over 30 popular public APIs across various categories including utility, finance, media, geo, and entertainment.

Execution: We used a Node.js script utilizing native http and https modules to send GET requests to each API endpoint. No caching layers were used on the client side.

Measurement: For each API, we made 5 consecutive requests with a strict 1-second delay between requests to avoid triggering aggressive rate limits. Latency was measured from request initiation to the completion of the response stream.

Metrics: The Average is the mean latency of all successful requests within a 30-second timeout window. Any requests resulting in network errors or timeouts were excluded from the average calculation, but their failed status is reflected in the dataset.

Environment: Tests were executed from a cloud server environment. Real-world latency from end-user devices will vary based on geographic location, ISP routing, and local network conditions.
