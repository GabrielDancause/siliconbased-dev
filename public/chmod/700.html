When to use chmod 700

Use 700 for: your home directory, private scripts, directories containing sensitive data, SSH directories (~/.ssh), backup directories, and any executable that only you should run.

Security note: 700 is very secure. The only risk is that it might be too restrictive — services running as other users (like a web server) won't be able to access the files at all.

Consider instead: If a service needs to read files: use 755 or 750. If you need group access: use 770. For non-executable private files: use 600.

Common Mistake

Setting web server directories to 700 when the web server runs as a different user (www-data, nginx). The server can't serve files it can't read. You need 755 for web-accessible directories.

chmod 700 vs 770

700 vs 770
700 vs 770: 770 gives group members the same full access as the owner while still blocking others. Use 770 for shared project directories where a team needs access but outsiders don't.

Frequently Asked Questions

Should my home directory be 700 or 755?
700 is more secure and recommended. With 755, other users can list your files and enter your directories. Most modern Linux distributions default home directories to 700.
Should .ssh be 700?
Yes, ~/.ssh must be 700. SSH will refuse to use keys if the directory has more permissive access. This is a hard requirement, not just a best practice.
What about 600 vs 700?
The difference is execute permission. Use 700 for directories (you need execute to enter them) and executable scripts. Use 600 for regular files like keys and configs.
