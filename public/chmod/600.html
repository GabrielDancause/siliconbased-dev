When to use chmod 600

Use 600 for: SSH private keys, .env files, config files with passwords or API keys, database credential files, SSL private keys, personal documents, and any file containing secrets.

Security note: 600 is one of the most secure file permissions. It ensures only the file owner can access the content. The only consideration is making sure the correct user owns the file.

Consider instead: If the file is a script that needs to run: use 700. If a group needs to read it: use 640. If it's a public file: use 644.

Common Mistake

Forgetting that SSH requires 600 on private keys. If your id_rsa file is 644, SSH will display a warning and refuse to use it. Always: chmod 600 ~/.ssh/id_rsa.

chmod 600 vs 644

600 vs 644
600 vs 644: The difference is that 644 allows everyone to read the file. For sensitive files (keys, passwords, configs), always use 600. For public files (web pages, images), use 644.

Frequently Asked Questions

Why does SSH require 600 on private keys?
SSH enforces strict permissions to prevent other users from stealing your private key. If the key is readable by others (e.g., 644), SSH considers it compromised and refuses to use it.
Should .env files be 600?
Yes. .env files typically contain database passwords, API keys, and secrets. With any permission more open than 600, other users on the server could read your credentials.
What about 400 vs 600?
400 is read-only even for the owner. Use 400 for files you want to protect from accidental modification (like deployed SSL certificates). Use 600 when you need to edit the file.
