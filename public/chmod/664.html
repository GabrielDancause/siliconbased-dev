When to use chmod 664

Use 664 for: shared config files, team documentation, collaborative source code files on a shared server, and any non-executable file that multiple team members need to edit.

Security note: 664 is generally safe. The main risk is that any user in the group can modify the file. Ensure group membership is intentional.

Consider instead: If only the owner should edit: use 644. If others shouldn't read: use 660. If the file is executable: use 775.

Common Mistake

Using 664 on a web server where the web server user is in the same group. This could allow a compromised web application to modify config files. Separate the web server into its own group.

chmod 664 vs 644

664 vs 644
664 vs 644: The only difference is group write permission. 644 = only the owner writes. 664 = the owner and group write. Use 664 for shared team files, 644 for single-owner files.

Frequently Asked Questions

What's the umask for 664?
A umask of 002 produces 664 for files and 775 for directories. This is common on systems set up for group collaboration. Check with: umask
Is 664 good for git repositories?
Git manages its own permissions, but setting core.sharedRepository = group in git config will create files as 664 in bare repositories, enabling team push access.
Should source code files be 664?
Only on shared servers where multiple developers edit files directly. In modern workflows with git, each developer works on their own copy (644) and shares via commits, not file permissions.
